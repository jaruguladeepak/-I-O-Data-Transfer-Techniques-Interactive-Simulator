Core Concepts

How Each Technique Works

Three fundamentally different approaches — same hardware, wildly different CPU utilization.

🔄

Programmed I/O

Polling / Busy-wait

The CPU continuously reads the device status register in a tight loop until it signals READY. Every CPU cycle is "wasted" — zero other work is possible.

  • CPU utilization: 100%
  • Zero interrupt overhead
  • Simple hardware
  • Worst for large transfers
Overhead = N_blocks × Poll_Check_Cycles
CPU Util = 100%
⚠ Poor CPU Efficiency
⚡

Interrupt-Driven I/O

Event-driven ISR

CPU issues a read command and switches to other tasks. The device fires an interrupt when each block is ready; CPU handles the ISR, then resumes.

  • CPU freed between transfers
  • ISR overhead per block
  • Good for moderate data
  • N interrupts for N blocks
ISR Cost = N × (Entry + Body + Exit)
CPU Util ≈ ISR fraction only
✓ Good CPU Efficiency
🚀

Direct Memory Access

Bus-Mastering DMA

CPU programs the DMA controller (source, destination, count) then is completely free. DMA moves all data autonomously and fires one completion interrupt.

  • CPU utilization: ~0%
  • Single completion interrupt
  • Best for large transfers
  • Fixed setup cost
Overhead = DMA_Setup + Completion_IRQ
CPU Util ≈ ~0%
🏆 Best CPU Efficiency

Exam Prep

Viva Preparation Guide

Key questions your professor is likely to ask — with clear, precise answers.

Q1
Why does DMA win for large data transfers?
DMA removes the CPU from the data path entirely using bus mastering. The CPU only pays a one-time setup cost and handles a single completion interrupt — unlike Interrupt I/O which pays ISR overhead every block.
Q2
Why is Polling CPU utilization exactly 100%?
During polling the CPU executes: while (status != READY) check(); — it cannot execute any other instruction. This busy-wait loop consumes 100% of CPU cycles reading a status register.
Q3
When should you choose Interrupt over DMA?
When transferring below the break-even point: N_break = ceil(DMA_setup / ISR_per_block). For tiny transfers, DMA's fixed setup cost exceeds the cumulative ISR overhead of Interrupt I/O.
Q4
What overhead does each ISR introduce?
(1) Entry — save CPU registers to stack (~100 cycles) · (2) Body — copy data to memory · (3) Exit — restore registers (~100 cycles). This context-switch overhead accumulates once per block.
Q5
What is cycle stealing in DMA?
The DMA controller becomes bus master during transfer. It steals bus cycles from the CPU to move data. The CPU may briefly stall waiting for the bus, but otherwise executes freely from cache.
Q6
Calculate the break-even point mathematically.
ISR_per_block = Entry + Body + Exit
DMA_total = DMA_Setup + Completion_IRQ
DMA wins when: N × ISR_per_block > DMA_total
Break-even: N_break = ⌈ DMA_total / ISR_per_block ⌉
Q7
How is throughput calculated?
Throughput = Total_Bytes / Transfer_Time
Polling transfers with CPU in the critical path so effective throughput ≤ device speed. DMA achieves near-peak device throughput since the CPU never bottlenecks the transfer.
Q8
How is CPU utilization calculated?
CPU_Util = (Busy_Cycles / Total_Cycles) × 100%
For Polling: all cycles are busy. For Interrupt: only ISR cycles are busy. For DMA: only setup + completion IRQ cycles are busy — typically <1% of total.
